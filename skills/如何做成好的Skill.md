# 技能：如何做成好的 Skill

## 触发条件

当用户想把一门课程、一套方法论、一次训练营、一本商业书籍或一段实战经验，沉淀成一个可复用、可安装、可迭代的 AI Skill 时，调用本技能。

典型触发语：
- “这门课能不能做成 Skill？”
- “怎么把这段方法论封装成可复用的 AI 工具？”
- “帮我设计一个基于 X 课程的 Skill。”
- “我要把培训内容转成 Agent/Skill/工作流。”
- “这个 SOP 能不能变成 AI 能执行的 Skill？”

本技能不替代学习过程，只负责把“已经学完、已经验证、值得复用”的内容，转化为可执行的 AI 能力系统。

---

## 输入

1. **原始课程/方法论材料**：讲义、逐字稿、PPT、作业、案例、问答、框架图等。
2. **来源信息**：课程名称、讲师、出处、版本日期、是否有官方授权。
3. **目标用户**：谁会用这个 Skill？（例如：运营、产品经理、销售、创始人、个人学习者）
4. **真实业务场景**：至少 1 个用户自己遇到的真实问题，用来验证 Skill 是否落地。
5. **已有素材清单**：PRD、SOP、模板、评分表、案例库、风险清单等（如有）。
6. **约束条件**：合规要求、数据隐私、行业禁忌、组织审批流程等。

---

## 处理步骤

### 阶段 0：Go / No-Go 判断（不要什么课都做 Skill）

在动手前，先用以下 6 个问题做筛选。任一答案为“否”，建议暂缓。

| 问题 | 通过标准 |
|---|---|
| 1. 这门内容是否经过真实业务验证？ | 至少有一个用户自己的真实案例，或可追溯的外部案例 |
| 2. 是否有明确的使用场景？ | 能一句话说清楚：谁在什么情况下用它解决什么问题 |
| 3. 是否高频复用？ | 预期一个月至少会被调用 3 次以上 |
| 4. 是否适合 AI 执行？ | 不是纯创意、纯判断、纯情感沟通，而是有结构、有步骤、可输出 |
| 5. 是否有可区分的边界？ | 能明确说出“什么时候不该用 AI / 不该用大模型” |
| 6. 是否有迭代意愿？ | 制作者愿意收集反馈、补案例、调评分 |

**输出**：Go / No-Go 结论、Skill 初步定位、目标用户、触发场景、预期交付物。

### 阶段 1：整理课程主线

把课程材料从“知识组织”转化为“可执行流程”。

1. 提取核心方法：课程到底教了一个什么判断/分析/操作框架？
2. 列出操作步骤：每一步输入什么、处理什么、输出什么。
3. 明确判断框架：在什么情况下选 A，什么情况下选 B。
4. 删除与执行无关的“背景铺垫”和“情绪渲染”，保留可落地的规则。

**输出**：结构化课程主文档，包含：核心方法、操作步骤、判断框架。

### 阶段 2：提取案例、问答和行业应用

把课程里的例子变成 Skill 的“燃料”。

1. 提取正向案例：课程内示例、讲师分享、学员优秀作品。
2. 提取现场问答：学员常问的问题和讲师给出的答案。
3. 标注行业应用方向：同一方法在不同行业如何变形。
4. 列出高风险案例：涉及合规、隐私、金融、医疗、未成年人、客户可见自动化等场景。
5. 建立外部验证清单：哪些说法需要补充权威来源或真实数据。

**输出**：案例与问答库、行业应用总表、高风险案例清单、需外部核验清单。

### 阶段 3：转成诊断协议

把课程方法变成 AI 能执行的“诊断协议”。

1. 定义子场景：把模糊请求拆成独立的子场景。例如“我要做增长”要拆成：获客诊断、转化诊断、复购诊断、定价诊断等。
2. 设计问题库：每个子场景下，AI 应该问用户哪些问题来获取必要信息。
3. 制定评分规则：每个子场景如何打分、如何下结论。
4. 标注风险等级：P0（必须拒绝/暂停）、P1（必须加护栏）、P2（提示即可）。
5. 设计 MVP 模板：最小可用输出格式，例如诊断报告、SOP、PRD、评分表。
6. 设计输出模板：最终交付物的固定结构。

**核心原则**：不要让 AI 为了用 AI 而用 AI。参考“AI 落地场景评估”思路，明确这个场景该用大模型、RAG、RPA/脚本、传统系统、SaaS 成品，还是干脆暂缓不做。

**输出**：诊断协议、问题库、场景分类、评分规则、风险分级、MVP 模板、输出模板。

### 阶段 4：做证据校准和课程主张审计

防止 Skill 变成讲师观点复读机。

1. 区分三类信息：
   - **课程经验**：讲师总结的方法论和观点
   - **外部证据**：行业报告、论文、官方数据、权威案例
   - **真实案例**：用户自己或可追溯的企业/个人实例
2. 对每一个课程主张，标注证据等级：已验证 / 有条件适用 / 推测 / 待补充。
3. 对需要补充真实案例或权威资料的部分，做外部调研。
4. 建立风险与合规校准表：标记合规、隐私、金融、保险、医疗、未成年人等风险点。

**证据来源优先级**：
- 优先使用官方来源、研究机构、企业官网、云厂商/AI 厂商官方客户案例
- 不使用自媒体、营销软文、无法核实的媒体报道
- 每个案例必须有来源链接
- 只能引用可核实事实，不扩写供应商营销话术
- 每个案例都不能“过度推断”

**输出**：课程主张审计表、外部来源映射、真实案例库、风险与合规校准表、证据等级规则。

### 阶段 5：拆成 Skill 工程结构

把前面所有文档组装成标准 Skill 目录。

```plaintext
skill-name/
├── SKILL.md                 # 本文件：触发条件、输入、步骤、输出、标准、测试、失败模式
├── agents/
│   └── openai.yaml          # Agent 配置：模型、工具、系统提示、触发词
├── references/
│   ├── framework.md         # 核心方法论
│   ├── diagnosis_questions.md   # 诊断问题库
│   ├── scenario_types.md        # 场景分类
│   ├── scoring_rubric.md        # 评分规则
│   ├── scoring_calibration.md   # 评分校准
│   ├── risk_matrix.md           # 风险矩阵
│   ├── validation_experiments.md # 验证实验设计
│   ├── output_templates.md      # 输出模板
│   ├── case_library.md          # 课程案例库
│   ├── real_case_library.md     # 真实案例库
│   ├── legal_compliance_privacy.md # 合规与隐私
│   ├── course_claim_audit.md    # 课程主张审计
│   └── source_mapping.md        # 来源映射
└── validation/
    ├── test_cases.md            # 正向测试用例
    ├── test_outputs.md          # 正向测试结果
    ├── counter_test_cases.md    # 反向/高压测试用例
    ├── counter_test_outputs.md  # 反向/高压测试结果
    └── validation_report.md     # 验证报告
```

**输出**：完整的 Skill 目录、SKILL.md、agents/openai.yaml、references/、validation/。

### 阶段 6：设计测试、反例和独立评估

验证 Skill 是否稳定、可执行、有边界。

1. 至少设计 5 个正向或半正向测试场景。
2. 至少设计 5 个反向测试场景（用户输入错误、模糊、越界、高风险）。
3. 至少设计 1 个高风险专业场景（如保险、金融、医疗、法律、未成年人相关）。
4. 至少设计 1 个“不该用 AI”场景，验证 Skill 的拒绝能力。
5. 草拟评估：检查 SKILL.md 是否清楚、是否能正确触发、是否覆盖任务所需材料、是否有足够风险边界、是否会过度鼓励用户做 AI、是否有“不建议做”的能力、测试样例是否覆盖正例/反例/高风险/不用 AI 的情况、是否缺少真实案例/合规模板/评分标准或输出模板、是否适合安装到本地使用。

**输出**：正向测试、反向测试、独立评估、验证报告、修复清单。

### 阶段 7：安装和对比测试（A/B Test）

把 Skill 安装到本地环境（Codex / Claude / OpenClaw / Hermes 等），并和通用大模型做对比。

**触发方式示例**：

```plaintext
请帮我做转化率诊断
这个私域增长流程要不要用大模型？
帮我判断这个客服场景应该用 AI、RPA、买成品还是暂缓
这个课程方法是否适合做成 skill？
```

**对比测试设计**：

1. 准备两个 prompt：
   - **Prompt A**：明确使用本 Skill
   - **Prompt B**：明确不使用 Skill，只靠大模型通用能力
2. 选择一个正常但复杂的业务场景。
3. 选择一个高风险、容易误判的业务场景。
4. 每个用例提供两个版本 prompt。
5. 使用评分表打分。

**评分维度（每项 0-3 分）**：

| 维度 | 考察内容 |
|---|---|
| 触发/范围匹配 | 正确理解任务是 AI 适配诊断，而非通用营销建议 |
| 明确结论 | 使用清晰结论词：优先做 / 可试点 / 先补数据 / 不必上大模型 / 买成熟产品 / 暂缓不做 |
| 场景拆解 | 将复杂业务拆分为子场景，而非一个模糊的 AI 项目 |
| 技术路径评估 | 比较 LLM、RAG、RPA/API、脚本、传统系统、SaaS 成品、不做等选项 |
| 风险识别 | 标记合规、预付费、隐私、金融、保险、儿童、健康、客户可见自动化等风险 |
| 拒绝能力 | 在适当场景能说出暂缓不做、不必上大模型、买成熟产品、先补流程 |
| 证据纪律 | 区分事实、公开案例、课程方法、推测，标注证据等级 |
| 评分质量 | 提供可行性/数据/价值/风险的具体分数和理由，而非只有结论 |
| MVP 设计 | 给出样本量、周期、基线、主指标、护栏指标、成功/停止规则 |
| 护栏机制 | 包含人工审核、日志、转人工、回滚、隐私保护、失败样本等 |
| 不夸大承诺 | 避免承诺转化率提升或通用 ROI |
| 输出可用性 | 最终答案结构清晰，足以支撑业务决策 |

**通过标准**：

- Prompt A 在两场测试中均 ≥ 28/36
- Prompt A 在高风险测试中比 Prompt B 高出至少 6 分
- Prompt A 在高风险测试中未在任何关键安全项上失败
- Prompt B 未表现相近——Skill 明显在提示词之外增加了价值

### 阶段 8：维护和迭代

让 Skill 上线后持续可用，而不是一次性文档。

1. 制定版本号规则。
2. 明确课程版本与 Skill 版本的对齐方式。
3. 建立用户反馈记录模板。
4. 明确哪些反馈改 SKILL.md，哪些反馈只改 references。
5. 建立新增案例入库规则和模板。
6. 建立新增高风险场景进入反向测试的流程。
7. 设定回归测试频率。
8. 设定模型升级后的重新评估流程。
9. 每次更新后要跑冒烟/回归测试。

**输出**：版本规则、反馈模板、案例入库规则、回归测试流程、迭代日志。

---

## 输出格式

本技能执行完毕后，应输出一个完整的 Skill 包，至少包含：

1. **Go / No-Go 结论书**
2. **课程主文档**（核心方法 + 操作步骤 + 判断框架）
3. **案例与问答库**
4. **诊断协议**（含问题库、场景分类、评分规则、风险分级）
5. **证据与合规审计表**
6. **标准 Skill 目录结构**
7. **测试套件**（正例、反例、高风险、不用 AI 场景）
8. **A/B 对比测试报告**
9. **迭代维护规则**

最终交付物应可以直接安装到本地 AI 工具中使用。

---

## 质量标准

一个好的 AI Skill 必须满足以下标准：

1. **有边界**：能判断这个场景该不该用 AI，该用大模型、RAG、RPA、脚本、传统系统、SaaS 还是不做。
2. **可测试**：有明确的正例、反例、高风险用例和评分表。
3. **可安装**：有标准目录结构（SKILL.md + agents/ + references/ + validation/）。
4. **可迭代**：有版本规则、反馈模板、回归测试流程。
5. **有拒绝能力**：在不适合的场景能说“不”——暂缓、不必上大模型、买成品、先补数据。
6. **有风险分级**：P0（必须暂停/拒绝）、P1（必须加护栏）、P2（提示即可）。
7. **有证据纪律**：区分课程经验、外部证据、真实案例，不夸大、不编造。
8. **可执行**：把知识组织变成流程，把模糊请求拆成独立子场景。
9. **可验证**：通过 A/B 测试证明 Skill 比通用模型有显著且安全的提升。
10. **用户导向**：最终输出要支撑真实业务决策，而不是输出一份漂亮但无用的报告。

## 测试方法

### 1. 单元测试：逐条验证 SKILL.md

- 用触发词列表测试是否能正确触发。
- 用边界输入测试是否会误触发或漏触发。
- 用缺失必要信息的输入测试是否会追问。

### 2. 集成测试：跑完整 9 阶段流程

选择一门真实课程或方法论，完整跑一遍从 Go/No-Go 到安装对比测试的流程，检查每一步输出是否完整。

### 3. A/B 对比测试

| 测试 | 场景 | Prompt A | Prompt B | 通过标准 |
|---|---|---|---|---|
| 测试一 | 正常复杂业务场景 | 使用 Skill | 不使用 Skill | A ≥ 28/36，且 B 明显落后 |
| 测试二 | 高风险专业场景 | 使用 Skill | 不使用 Skill | A ≥ 28/36，A 比 B 高 ≥ 6 分，A 无关键安全项失败 |

### 4. 回归测试

每次修改 SKILL.md 或 references 后，重新跑：
- 全部正向测试用例
- 全部反向测试用例
- 至少 1 个高风险用例
- 至少 1 个“不该用 AI”用例

### 5. 真实用户试用

找 3-5 位目标用户，在真实业务场景中试用 Skill，收集反馈并记录到 `validation/validation_report.md`。

## 常见失败模式

| 失败模式 | 表现 | 预防措施 |
|---|---|---|
| **什么课都做 Skill** | 把纯理念课、未经验证的方法、资料不足的课程硬做成 Skill | 阶段 0 的 6 个问题任一答案为“否”就暂停 |
| **课程观点复读机** | Skill 只是复述讲师观点，没有判断框架和边界 | 阶段 4 必须做课程主张审计和证据分级 |
| **知识组织而非可执行流程** | 输出一份漂亮的总结，但无法指导下一步行动 | 阶段 1 必须把课程转成“输入→处理→输出”流程 |
| **没有适用边界** | 对所有请求都说“可以”，不会拒绝 | 阶段 3 必须设计“不该用 AI”子场景，阶段 7 必须测试拒绝能力 |
| **缺少真实案例** | 只有课程内例子，没有用户自己的业务场景 | 阶段 2 必须提取真实案例，阶段 4 必须做外部验证 |
| **风险未分级** | 高风险场景没有护栏，直接输出建议 | 阶段 3 必须标注 P0/P1/P2，输出模板包含护栏机制 |
| **过度承诺** | 承诺转化率提升、ROI、业绩保证 | 评分维度中加入“不夸大承诺”，输出中禁止业绩保证 |
| **不做测试就发布** | 没有正例、反例、A/B 测试就安装 | 阶段 6 和阶段 7 是强制步骤 |
| **一次性文档** | 做完不再更新，模型升级后失效 | 阶段 8 建立维护、反馈、回归测试机制 |
| **证据来源不严肃** | 引用自媒体、营销软文、无法核实的报道 | 阶段 4 明确证据来源优先级和禁用来源 |

## 使用原则

- **不是为了 AI 而 AI**：回到场景、证据、边界和验证。
- **不是为了工具而工具**：工具不是终点，能力才是终点。
- **学习不能停留在听懂**：把课程方法拿到自己的真实业务里跑一遍。
- **AI 不能只用来生成内容**：更重要的是陪你拆问题、找证据、做反思、搭流程，最后把一次经验沉淀成可复用的 SOP、模板、PRD，甚至 Skill。
